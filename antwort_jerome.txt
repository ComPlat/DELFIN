danke fuers Log - das war aufschlussreich. Entwarnung vorweg: der DELFIN-Agent
ist nicht abgestuerzt. Der Fehler steckt im Porpoise-Dashboard selbst.

Was das Log zeigt:

- Der eigentliche Fehler (ganz unten):
  UnboundLocalError: cannot access local variable 'on_file_upload_change'
  in porpoise/dashboard/__init__.py:533 (_create_tab_container).
  Das ist ein Python-Bug im Porpoise-Code: die Funktion on_file_upload_change
  wird benutzt, BEVOR sie definiert ist (oder nur in einem if-Zweig definiert,
  der nicht lief).

- Die vielen Nachrichten ("Kernel started" im Dutzend):
  Das ist Voilas Kernel-Restart-Schleife - die Dashboard-Init-Zelle crasht ->
  Voila startet den Kernel neu -> Zelle crasht wieder -> usw. Ein Crash-Loop,
  ausgeloest vom obigen Bug.

- Die "403 ... '_xsrf' argument missing"-Zeilen:
  harmlos - das sind nur deine Ctrl-C-Shutdown-Versuche, die Voila ohne
  XSRF-Token ablehnt. Kein eigenes Problem.

Warum VS Code instabil wurde: sehr wahrscheinlich die FOLGE des Crash-Loops -
Dutzende Kernel-Neustarts spawnen je einen schweren Python-Prozess (mit allen
Imports) -> CPU/RAM-Spike -> der Remote-Server wird instabil/OOM. Nicht der
Agent, sondern die ungebremste Restart-Schleife auf der crashenden Zelle. Dass
VS Code diesmal angemeldet blieb, passt dazu: es war die Voila-/Kernel-Ebene,
nicht VS Code selbst.

Der Fix (in Porpoise, nicht DELFIN): In porpoise/dashboard/__init__.py die
"def on_file_upload_change(...)" VOR ihre erste Verwendung ziehen - bzw.
sicherstellen, dass sie immer definiert wird, nicht nur in einem Zweig.
Zwei-Minuten-Sache.
